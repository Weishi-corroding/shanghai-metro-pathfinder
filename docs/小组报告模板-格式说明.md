# 《数据结构课程设计——小组设计报告》模板格式说明

> 本文档基于 `小组报告模板.docx` 提取整理，详尽描述模板的格式、字体、字号、内容结构与批注要求，用于指导撰写类似的小组报告。

---

## 一、页面设置

| 属性 | 值 |
|---|---|
| 纸张大小 | A4 (210mm × 297mm / 8.3 × 11.7 英寸) |
| 页边距-上下 | 1.0 英寸 (25.4mm) |
| 页边距-左右 | 1.2 英寸 (30.5mm) |
| 纸张方向 | 纵向 (Portrait) |
| 节数 | 3 节（封面、正文各节页面设置相同） |

## 三、文档整体结构

模板正文共 4 章，结构如下：

```
封面页
  ├── "数据结构课程设计"（36pt, 宋体, Bold, 居中）
  ├── "----小组设计报告"（22pt, 宋体, Bold, 居中）
  ├── 组号：___（下划线填写）
  ├── 指导教师：___（下划线填写）
  ├── 组成员及贡献度表格
  └── 日期：YYYY年M月D日至YYYY年M月D日（下划线填写）

目录（自动生成，含 TOC 1-3 三级）

1 课程设计目的                          ← Heading 1
  └── 5 条目的说明（Normal 正文列表）

2 任务完成情况                          ← Heading 1
  └── 表1：任务完成情况表（√/× 标记）

3 设计报告                              ← Heading 1
  ├── 3.1、问题描述                     ← Heading 2
  ├── 3.2、概要                         ← Heading 2
  │     ├── 表2：功能模块设计交互过程
  │     ├── 表3：代码实现交互过程
  │     └── 表4：测试与调试阶段交互过程
  ├── 3.3、功能模块设计                 ← Heading 2
  │     └── Copilot 交互（提问→回复→说明）
  ├── 3.4、代码实现                     ← Heading 2
  │     └── 多次 Copilot 交互，含代码清单
  ├── 3.5、使用示例                     ← Heading 2
  │     └── 运行截图 + 小标题说明
  └── 3.6、测试与调试                   ← Heading 2
        └── 3.6.1 bignum大数模块测试    ← Heading 3
              └── 测试用例表（序号、测试目的、输入数据、期望结果、实际结果、结论）

4 体会与感想                            ← Heading 1
  └── 自由文本段落
```

## 四、内容要求（按章节）

### 4.1 封面

- **组号**：二位数字（如 01）
- **指导教师**：教师姓名
- **组成员及贡献度**：表格形式，列出每位成员的**姓名**、**学号**、**贡献度**（数值，如 1.1、1、0.95，合计等于小组人数）
- **日期**：课程设计起止日期（格式：YYYY年M月D日至YYYY年M月D日）

### 4.2 目录

- 应使用 Word 自动目录域（TOC field），而非手动输入目录内容
- 域代码：`TOC \o "1-3" \h \z \u`（根据 Heading 1-3 三级标题自动生成）
- 包含页码，右对齐
- 生成 docx 时插入域代码，用户打开后在 Word 中右键"更新域"即可生成实际目录

### 4.3 第1章：课程设计目的

- 列出 5 条目的（模板提供标准条目）：
  1. 学习获取知识的方法
  2. 提高发现问题、分析问题和解决实际问题的能力
  3. 加强创新意识和创新精神
  4. 加强团队的分工与合作
  5. 掌握面向实际背景思考问题的方法
- 格式：Normal 正文，每条一句，无编号

### 4.4 第2章：任务完成情况

- 简要介绍任务完成情况
- **表1：任务完成情况表**，包含三列：
  - 任务序号
  - 任务名称（基本功能/进阶功能，细分具体功能点）
  - 完成情况（√ 完成 / × 未完成）
- 表1 即完成本章内容，无需额外统计文字

### 4.5 第3章：设计报告（核心章节）

#### 3.1 问题描述
- 用 1-2 句话描述项目要解决的问题
- 模板示例："有一个合数n，大小达到10进制的1000位。请编程将其拆分为两个素数p和q，使得p*q=n。"

#### 3.2 概要
- 概述整个项目开发过程包含三个步骤：
  1. 功能模块设计
  2. 代码实现
  3. 测试与调试
- 每个步骤需说明包含几轮交互，并概述交互过程
- 使用**表2/表3/表4**记录各阶段的交互过程，表格格式：轮次 | 提问 | 回复 | 思考及行动

#### 3.3 功能模块设计
- 内容要求：
  1. 向 Copilot 提问的原始内容（用 **【提问】：** 标记）
  2. Copilot 的回复内容（用 **【回复】：** 标记）
  3. 对回复的分析和行动（用 **【说明】：** 标记）
- 分析环节应包含：模块划分是否合理、是否需要合并/拆分、最终模块结构
- 画出**系统功能模块设计图**（图1），标出各模块及调用关系
- 列出模块间调用关系（使用 List Paragraph 样式）

#### 3.4 代码实现
- 由于模块较多，通过**多次提问**逐步实现
- 每次交互包含：
  - **【提问N】：** 明确、具体的编程要求
  - **【回复N】：** Copilot 生成的代码
  - **【说明】：** 对代码的分析，包括：
    - 是否看懂算法（看不懂的算法需画流程图）
    - 代码是否正确（发现 bug 要明确指出）
    - 如何将代码集成到项目中（文件创建步骤）
- **代码块格式**：Normal 样式，左对齐，不缩进，使用等宽展示（但模板实际用 Normal+宋体展示代码）
- **流程图要求**（红色批注特别强调）：
  - 复杂算法（如大数加减乘除）需画流程图以证明理解
  - 简单代码无需画流程图
  - 推荐使用 https://www.drawio.com/ 绘制
  - 也可手绘后拍照插入
  - 流程图编号：图2、图3……
- **文件结构图**：展示项目文件组织（图3）
- **修改后代码**：如修复了 bug，需展示修改后的完整代码

#### 3.5 使用示例
- 根据用户界面菜单功能逐一演示
- **关键要求**：将运行结果截图，加上小标题，让读者能清晰看出是哪个功能
- 每个功能一个截图 + 说明文字

#### 3.6 测试与调试
- 使用 Heading 3 为每个模块创建小节（如 3.6.1 bignum大数模块测试）
- **每个函数单独测试**，测试步骤模板：
  1. 说明测试目的和测试方法（为何将某两个函数一起测）
  2. 给出测试用的 main 函数代码
  3. 说明测试数据范围/约束
  4. **测试用例表**（表5-表11等），格式固定为 6 列：
     | 序号 | 测试目的 | 输入数据 | 期望结果 | 实际结果 | 结论 |
  5. 分析测试结果：成功/失败的原因
  6. 如发现 bug → Copilot 修复交互（【提问】→【回复】→修改后代码）→ **全部重新测试**（强调：不能只测原来失败的用例）
- 测试用例设计原则：
  - 正常情况
  - 边界情况（0、1、最大值、最小值）
  - 特殊情况（前导0、进位/借位、等值比较）
- 至少覆盖：intToBigNum、strToBigNum、isZero、cmpBigNum、addBigNum、subBigNum、mulBigNum、divBigNum、modBigNum、isPrime、main 分解逻辑

### 4.6 第4章：体会与感想

- 自由撰写，内容包括但不限于：
  - 编程能力的提高
  - 遇到的困难及解决方法
  - 对算法实现的理解
  - 团队协作的体会
  - 展望与不足
- 格式：Normal 正文段落（模板中注明"仅供参考，请根据内容填写"）

## 五、批注与标注约定

### 5.1 红色文字（教师批注）

模板中以下内容以**红色字体 (RGB: FF0000)** 标注，属于模板制作者的指导性说明：

> 出现在 3.4 代码实现部分的流程图指导：
> - "这里可对一些复杂的算法画流程图，简单的不用画。本项目中，大数的加、减、乘、除四个算法可以画流程图，以此表明自己看明白了算法。"
> - "画流程图推荐使用 https://www.drawio.com/，如果实在不熟悉，也可以在纸上画，然后拍照放进word文档里。"
> - "以下是用drawio画的流程图示例，与加法算法无关"

**含义**：红色文字是给撰写报告者的提示，最终报告应删去红色文字并替换为实际内容。

### 5.2 占位符/待替换标记

模板中使用以下占位符，需替换为实际内容：
- `**` — 表示需填写具体数字（如交互轮数）
- `……` — 表示需补充具体内容
- `（此处省略n字）` — 表示类似内容可照此模式继续
- `（以下省略n字）` — 表示章节可照已有模式扩展
- `（请根据内容填写）` / `（请根据实际完成情况填写）` — 提示需自定义

### 5.3 交互标记格式

整个第3章的核心格式是 Copilot 交互记录：

```
【提问N】：<具体的编程问题或要求>
【回复N】：<Copilot生成的完整回复，包含代码>
【说明】：<对回复的分析、代码检查、bug发现、修改要求、文件操作步骤>
```

- **提问**和**回复**必须如实记录
- **说明**是体现学生思考和理解的关键部分，应包含：
  - 代码正确性分析
  - 发现的 bug 及其原因
  - 纠正 bug 的具体指示
  - 文件创建/修改的操作步骤
  - 算法理解（看不懂的配流程图）

## 八、代码展示格式

- 使用 Normal 样式（10.5pt 宋体），左对齐
- 保持原始缩进（用空格缩进）
- 包含必要的注释
- 修改前后的代码都应完整展示
- 关键修改处可用加粗或其他标记标注
